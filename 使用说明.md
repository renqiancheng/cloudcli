# CloudCLI UI 使用说明

本文档面向项目开发者，说明 CloudCLI UI（`@cloudcli-ai/cloudcli`）的技术栈、架构设计、本地调试运行方式、生产环境持久部署（PM2）以及发布流程。

---

## 1. 项目概述

CloudCLI UI 是一个 Web 端 UI，用于管理和使用多种 AI 编码 Agent CLI：

- **Claude Code**（Anthropic）
- **Cursor CLI**
- **Codex**（OpenAI）
- **Gemini CLI**（Google）

项目采用前后端分离架构：React 前端 + Express 后端，通过 REST API 与 WebSocket 与各类 Agent CLI 交互，支持文件浏览、Git 操作、Shell 终端、会话管理、MCP 配置、插件扩展等功能。

- **npm 包名**：`@cloudcli-ai/cloudcli`
- **CLI 命令**：`cloudcli`
- **默认访问地址**：`http://localhost:3001`（生产模式）/ `http://localhost:5173`（开发模式）
- **许可证**：AGPL-3.0-or-later

## 3. 架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                        浏览器客户端                           │
│  React SPA (Vite) — 聊天 / 文件树 / Git / Shell / 设置       │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP REST (/api/*)
                           │ WebSocket (/ws, /shell, /plugin-ws)
┌──────────────────────────▼──────────────────────────────────┐
│                    Express 后端 (Node.js)                     │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────────┐  │
│  │ Routes   │  │ Middleware   │  │ WebSocket Module      │  │
│  │ (REST)   │  │ (Auth/CORS)  │  │ /ws /shell /plugin-ws │  │
│  └──────────┘  └──────────────┘  └───────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Providers Module (claude / cursor / codex / gemini)   │   │
│  │  auth · mcp · skills · sessions · sessionSynchronizer │   │
│  └──────────────────────────────────────────────────────┘   │
│  ┌──────────────┐  ┌──────────────┐  ┌─────────────────┐   │
│  │ Database     │  │ Projects     │  │ Plugin Manager  │   │
│  │ (SQLite)     │  │ Module       │  │                 │   │
│  └──────────────┘  └──────────────┘  └─────────────────┘   │
└──────────────────────────┬──────────────────────────────────┘
                           │ spawn / SDK
┌──────────────────────────▼──────────────────────────────────┐
│              本地 Agent CLI 进程                                │
│  claude · cursor · codex · gemini                             │
└───────────────────────────────────────────────────────────────┘
```

### 3.2 目录结构

```
cloudcli/
├── src/                    # React 前端源码
│   ├── components/         # UI 组件（chat、file-tree、git-panel 等）
│   ├── contexts/           # React Context
│   ├── hooks/              # 自定义 Hooks
│   ├── i18n/               # 多语言翻译文件
│   ├── types/              # 前端 TypeScript 类型
│   └── utils/              # 前端工具函数
├── server/                 # Express 后端
│   ├── index.js            # 服务入口
│   ├── cli.js              # CLI 命令入口
│   ├── routes/             # REST API 路由
│   ├── middleware/         # 中间件（认证等）
│   ├── modules/            # 模块化后端代码
│   │   ├── providers/      # Agent Provider 抽象层
│   │   ├── websocket/      # WebSocket 网关
│   │   ├── database/       # SQLite 数据层
│   │   └── projects/       # 项目管理
│   └── utils/              # 后端工具
├── shared/                 # 前后端共享代码
├── public/                 # 静态资源、PWA manifest
├── dist/                   # 前端生产构建输出
├── dist-server/            # 后端编译输出
├── docker/                 # Docker 沙箱镜像
├── scripts/                # 辅助脚本
└── .github/workflows/      # CI/CD 工作流
```

### 3.3 Provider 模块

每个 Agent CLI 在后端通过统一的 Provider 抽象接入，每个 Provider 包含五个能力面：

| 能力面 | 职责 |
|--------|------|
| `auth` | 检测 CLI 安装与认证状态 |
| `mcp` | 读写 Provider 原生 MCP 配置 |
| `skills` | 发现 Provider 原生 Skill 文件 |
| `sessions` | 标准化会话事件与历史记录 |
| `sessionSynchronizer` | 扫描本地会话文件并同步到数据库 |

当前支持的 Provider ID：`claude`、`cursor`、`codex`、`gemini`。

### 3.4 WebSocket 通信

后端 WebSocket 网关提供三条通道：

| 路径 | 用途 |
|------|------|
| `/ws` | 聊天流式通信，转发 Agent 命令与响应 |
| `/shell` | 交互式 PTY 终端（支持断线重连） |
| `/plugin-ws/:pluginName` | 插件 WebSocket 代理 |

开发模式下，Vite 会将 `/api`、`/ws`、`/shell` 代理到后端 `SERVER_PORT`（默认 3001）。

### 3.5 数据存储

| 数据 | 默认位置 |
|------|----------|
| 认证数据库 | `~/.cloudcli/auth.db` |
| Claude 会话 | `~/.claude/projects/` |
| Codex 会话 | `~/.codex/sessions/` |
| Cursor 会话 | `~/.cursor/projects/` |
| Gemini 会话 | `~/.gemini/tmp/` |
| 环境配置 | 项目根目录 `.env` |

## 4. 环境要求

### 4.1 必备

- **Node.js** 22 或更高版本（见 `.nvmrc`）
- **npm**（随 Node.js 安装）
- 至少安装并配置一种 Agent CLI（如 Claude Code）

### 4.2 可选

- **Git**（Git 面板功能）
- **Docker + sbx CLI**（Docker 沙箱模式，实验性）
- **PM2**（生产环境进程守护，见 [第 7 节](#7-生产环境持久运行pm2)）

### 4.3 原生模块编译

项目依赖 `better-sqlite3`、`node-pty`、`bcrypt` 等原生模块。`npm install` 时会自动执行 `postinstall` 脚本修复 node-pty。若安装失败，请确保已安装 C++ 构建工具链（Windows 上需要 Visual Studio Build Tools）。

## 5. 环境配置

### 5.1 创建 .env 文件

复制示例配置并按需修改：

```bash
cp .env.example .env
```

常用配置项：

```env
# 后端 API + WebSocket 端口
SERVER_PORT=3001

# 前端开发服务器端口（仅开发模式）
VITE_PORT=5173

# 绑定地址（0.0.0.0 允许局域网访问，127.0.0.1 仅本机）
HOST=0.0.0.0

# 认证数据库路径（可选，默认 ~/.cloudcli/auth.db）
# DATABASE_PATH=/path/to/auth.db

# Claude CLI 自定义路径（可选）
# CLAUDE_CLI_PATH=claude

# 上下文窗口大小
CONTEXT_WINDOW=160000
VITE_CONTEXT_WINDOW=160000
```

### 5.2 查看当前配置

```bash
npx cloudcli status
# 或在源码目录：
npm run server:dev -- status
```

## 6. 本地开发与调试

### 6.1 首次设置

```bash
# 克隆仓库
git clone https://github.com/siteboon/claudecodeui.git
cd claudecodeui

# 安装依赖
npm install

# 复制环境配置
cp .env.example .env
```

### 6.2 启动开发服务器（推荐）

同时启动前端 Vite 开发服务器和后端 Express 服务：

```bash
npm run dev
```

- 前端：`http://localhost:5173`（Vite 热更新）
- 后端：`http://localhost:3001`（API 与 WebSocket）
- Vite 自动将 `/api`、`/ws`、`/shell` 代理到后端

### 6.3 分别启动前后端

适用于只需调试某一端的场景：

```bash
# 仅后端（使用 tsx 直接运行 TypeScript/JS 源码）
npm run server:dev

# 后端热重载（文件变更自动重启）
npm run server:dev-watch

# 仅前端
npm run client
```

### 6.4 生产模式本地运行（前台）

适用于临时验证构建结果，关闭终端后服务会停止。若需持久后台运行，请使用 [第 7 节 PM2 方式](#7-生产环境持久运行pm2)。

```bash
# 构建前后端并启动
npm start

# 或分步执行
npm run build        # 构建前端 (dist/) + 后端 (dist-server/)
npm run server       # 启动编译后的后端，同时托管前端静态文件
```

访问 `http://localhost:3001`。

### 6.5 调试技巧

**后端调试**

- 使用 `npm run server:dev-watch` 实现后端热重载
- 后端编译产物带 Source Map（`server/tsconfig.json` 中 `sourceMap: true`）
- 可在 VS Code / Cursor 中对 `server/index.js` 或 `dist-server/server/index.js` 设置断点，使用 Node.js 调试器 attach

**前端调试**

- Vite 开发服务器自带 HMR，修改 `src/` 下文件即时生效
- 浏览器 DevTools → Network 面板查看 `/api` 请求
- 浏览器 DevTools → WS 面板查看 WebSocket 消息（`/ws`、`/shell`）

**CLI 命令**

```bash
cloudcli              # 启动服务（默认）
cloudcli start        # 启动服务
cloudcli status       # 查看配置与数据路径
cloudcli version      # 查看版本
cloudcli help         # 帮助信息
cloudcli sandbox      # Docker 沙箱管理（实验性）
```

### 6.6 代码质量检查

```bash
# TypeScript 类型检查（前端 + 后端）
npm run typecheck

npm run lint

# ESLint 自动修复
npm run lint:fix
```

## 7. 生产环境持久运行（PM2）

推荐使用 **构建生产版本 + PM2 进程守护** 的方式在本地正式运行。关闭终端后服务仍保持运行，崩溃时 PM2 会自动重启。

> **注意**：持久运行的是 Web UI 服务，不是 Agent 会话本身。Agent 仍依赖本机 CLI 及对应配置目录（如 `~/.claude`）。

### 7.1 构建生产版本

在项目根目录执行：

```bash
# 安装依赖（首次或 package.json 变更后）
npm install

# 复制并编辑环境配置（首次）
cp .env.example .env

# 构建前端 (dist/) + 后端 (dist-server/)
npm run build
```

构建成功后产物位于：

| 目录 | 内容 |
|------|------|
| `dist/` | 前端静态资源（HTML、JS、CSS） |
| `dist-server/` | 后端编译输出（含 Source Map） |

等价命令：

```bash
npm run build:client   # 仅构建前端
npm run build:server   # 仅编译后端
```

### 7.2 安装 PM2

```bash
npm install -g pm2
```

验证安装：

```bash
pm2 --version
```

### 7.3 启动服务

**方式 A：从源码目录启动（推荐，适合当前仓库）**

Windows 上 PM2 对 `npm run server` 传参支持不佳，建议直接启动编译后的入口文件，并指定工作目录以正确读取 `.env`：

```powershell
# PowerShell（Windows）
pm2 start dist-server/server/index.js --name "cloudcli-ui" --cwd "D:\Code\Github\cloudcli"
```

```bash
# Linux / macOS
pm2 start dist-server/server/index.js --name "cloudcli-ui" --cwd /path/to/cloudcli
```

**方式 B：全局安装后启动（适合日常使用）**

```bash
npm install -g @cloudcli-ai/cloudcli
pm2 start cloudcli --name "cloudcli-ui"
```

自定义端口：

```bash
pm2 start cloudcli --name "cloudcli-ui" -- --port 8080
```

启动后保存 PM2 进程列表（便于后续恢复）：

```bash
pm2 save
```

### 7.4 访问地址

| 模式 | 地址 | 说明 |
|------|------|------|
| 生产模式（PM2 / `npm run server`） | `http://localhost:3001` | 默认端口，以后端同时托管前端静态文件 |
| 开发模式（`npm run dev`） | `http://localhost:5173` | 仅开发调试用，不支持持久运行 |

局域网访问：在 `.env` 中设置 `HOST=0.0.0.0`，然后通过 `http://<本机IP>:3001` 访问。Windows 防火墙需放行对应端口。

查看当前配置与数据路径：

```bash
cloudcli status
```

### 7.5 PM2 常用命令

| 命令 | 说明 |
|------|------|
| `pm2 list` | 查看所有进程及运行状态 |
| `pm2 logs cloudcli-ui` | 查看实时日志 |
| `pm2 logs cloudcli-ui --lines 50 --nostream` | 查看最近 50 行日志（不持续跟踪） |
| `pm2 restart cloudcli-ui` | 重启服务 |
| `pm2 stop cloudcli-ui` | 停止服务 |
| `pm2 delete cloudcli-ui` | 从 PM2 中移除进程 |
| `pm2 save` | 保存当前进程列表 |
| `pm2 resurrect` | 恢复上次 `pm2 save` 保存的进程 |

### 7.6 代码更新后重启

修改源码或拉取新代码后，需重新构建并重启 PM2 进程：

```bash
cd /path/to/cloudcli    # 进入项目根目录

npm install             # 依赖有变更时执行
npm run build           # 重新构建生产版本
pm2 restart cloudcli-ui
```

全局安装方式更新：

```bash
cloudcli update
pm2 restart cloudcli-ui
```

### 7.7 设置开机自启

**Linux / macOS**

```bash
pm2 startup             # 输出一条需以 root 执行的命令，复制并运行
pm2 save                # 保存当前进程列表，重启后自动恢复
```

**Windows**

Windows 需额外安装启动工具：

```powershell
npm install -g pm2-windows-startup
pm2-startup install
pm2 save
```

重启电脑后，`cloudcli-ui` 进程会自动拉起。若未生效，可手动执行 `pm2 resurrect` 恢复。

### 7.8 注意事项

1. **不要用 `npm run dev` 做持久运行** — 开发模式关闭终端即停止，且前后端分离不适合生产部署。
2. **端口冲突** — 若 PM2 反复重启并出现 `EADDRINUSE`，说明 3001 端口已被占用。先查找并结束占用进程，再重启 PM2：

   ```powershell
   # Windows：查找占用 3001 端口的进程
   netstat -ano | findstr ":3001"
   # 结束对应 PID（将 <PID> 替换为实际值）
   taskkill /PID <PID> /F
   pm2 restart cloudcli-ui
   ```

3. **配置文件** — PM2 启动时需指定 `--cwd` 为项目根目录，确保 `.env` 被正确加载。
4. **认证数据** — 默认存储在 `~/.cloudcli/auth.db`，与 PM2 工作目录无关。

## 9. 发布流程

### 9.1 版本管理工具

项目使用 [release-it](https://github.com/release-it/release-it) 配合 `@release-it/conventional-changelog` 自动管理版本发布。配置见 `.release-it.json`。

### 9.2 本地发布（维护者）

**前提条件：**

- 当前分支为 `main`
- 工作区干净（无未提交变更）
- 已配置 `GITHUB_TOKEN` 和 npm 发布权限
- `.env` 中设置 `GITHUB_TOKEN=xxx`（`release.sh` 会读取）

**发布命令：**

```bash
# 交互式发布（提示选择版本号）
npm run release

# 指定版本增量
npm run release -- patch    # 补丁版本 (x.x.Z)
npm run release -- minor    # 次版本 (x.Y.0)
npm run release -- major    # 主版本 (X.0.0)
```

**release-it 自动执行：**

1. 运行 `npm run build` 确保构建通过
2. 根据 Conventional Commits 自动生成 `CHANGELOG.md`
3. 更新 `package.json` 版本号
4. 创建 Git 提交与 Tag（格式 `vX.Y.Z`）
5. 推送到 GitHub 并创建 GitHub Release
6. 发布到 npm（`@cloudcli-ai/cloudcli`，public access）

### 9.3 CI 自动发布

GitHub Actions 工作流 `.github/workflows/release.yml` 支持手动触发发布：

1. 进入 GitHub → Actions → **Release**
2. 点击 **Run workflow**
3. 选择版本增量（patch / minor / major 或具体版本号如 `1.33.0`）
4. 可选填写自定义 Release 名称

CI 会使用 `RELEASE_PAT` 和 `NPM_TOKEN` 密钥执行 release-it。

### 9.4 Docker 镜像发布

`.github/workflows/docker.yml` 支持手动构建并推送 Docker 沙箱镜像：

- 镜像仓库：`docker.io/cloudcliai/sandbox`
- 支持模板：`claude-code`、`codex`、`gemini`
- 多平台：`linux/amd64`、`linux/arm64`

### 9.5 提交规范

发布日志基于 [Conventional Commits](https://conventionalcommits.org/) 自动生成：

```
<type>(optional scope): <description>
```

| Type | 说明 | 版本影响 |
|------|------|----------|
| `feat` | 新功能 | minor |
| `fix` | Bug 修复 | patch |
| `perf` | 性能优化 | patch |
| `refactor` | 重构 | — |
| `docs` | 文档 | — |
| `chore` | 维护 | — |
| `ci` | CI/CD | — |
| `test` | 测试 | — |
| `build` | 构建 | — |

Breaking Change 使用 `feat!:` 或在 commit footer 中添加 `BREAKING CHANGE:`，触发 major 版本升级。

## 12. 常见问题

### Q: PM2 启动后反复重启，日志显示 `EADDRINUSE`？

3001 端口已被其他进程占用（可能是之前手动启动的 node 进程）。先结束占用进程，再执行 `pm2 restart cloudcli-ui`。详见 [7.8 注意事项](#78-注意事项)。

### Q: Windows 上 PM2 无法用 `npm run server` 启动？

Windows 下 PM2 对 npm 脚本传参支持有限，请改用直接启动编译产物：

```powershell
pm2 start dist-server/server/index.js --name "cloudcli-ui" --cwd "项目根目录绝对路径"
```

### Q: `npm install` 失败，提示 node-pty / better-sqlite3 编译错误？

确保已安装 C++ 构建工具。Windows 需安装 [Visual Studio Build Tools](https://visualstudio.microsoft.com/visual-cpp-build-tools/) 并勾选「使用 C++ 的桌面开发」。

### Q: 开发模式下前端能访问但 API 报错？

确认后端已启动（`npm run dev` 会同时启动）。检查 `.env` 中 `SERVER_PORT` 是否与 Vite 代理目标一致（默认 3001）。

### Q: 会话列表为空？

确认对应 Agent CLI 已安装并完成认证。CloudCLI 从本地 CLI 配置目录自动发现会话（如 `~/.claude/projects/`）。

### Q: 如何只允许本机访问？

在 `.env` 中设置 `HOST=127.0.0.1`。

### Q: 如何修改数据库位置？

在 `.env` 中设置 `DATABASE_PATH=/your/custom/path/auth.db`。
